# 合同数据程序化导入架构与管理模式

## 1. 结论

合同管理模块已推翻旧的 `sheet → 合同流` 设计。新版合同管理只围绕以下主线设计：

```text
Excel 宏观合同流骨架
→ Word 前向/后向合同文本解析
→ 前向合同 ↔ 后向合同 多对多候选关系
→ 人工确认
→ 设备级资金流单元
→ 业主设备级审计
→ 阶段级回款/付款释放
```

合同管理模块最终管理的不是“旧静态合同表”，而是：

```text
每一笔前向销售合同收入经过天安后，具体拆给哪些后向采购合同、哪些设备/软件/服务、哪些点位、哪些审计批次，以及哪些阶段可以回款和付款。
```

## 2. 不允许再使用的旧设计

以下设计已废弃，不得再接入合同管理功能页：

- 把 Excel 每个 sheet 当成一个合同。
- 把 sheet 名和合同流备注相同当成高置信度合同关系。
- 让用户确认“设备清单行是否绑定合同流”。
- 只展示静态旧静态合同表或静态统计卡片。
- 只管理合同级回款/付款。
- 把设备级付款当成一次性付款。
- 把前向付款阶段和后向付款阶段机械一一对应。
- 在候选关系未确认前进入应收、应付、毛利或付款审批计算。
- 把 `RSU`、`OBU`、`GPU`、`C-V2X` 当成合同主体。
- 继续使用旧名称 `旧的“吉利/希迪版本”表述`。

## 6. 合同管理功能页结构

新版合同管理功能页是：

```text
合同关系重构工作台
```

页面必须包含：

1. 宏观合同流骨架。
2. Word 合同匹配概览。
3. 前向合同 → 天安 → 后向合同多对多候选关系。
4. 设备级资金流单元。
5. 阶段付款释放看板。

### 6.1 宏观合同流骨架

展示 Excel `合同流` sheet 尾部识别出的宏观合同流，例如：

```text
车联网-天安（华通+交科所+隆顺）
移动-浪潮-天安（大为信号机+R1）
移动-上研-合创-天安（应用平台）
移动-上研-工业安装-天安（华通+公交硬件+R3等）
移动-中通服网盈-天安(R2)
移动-尚行-天安(车城+服务器）
移动-四维图新-天安（大华+海康智联+RSU)
```

宏观合同流可点击。选中后，应筛选右侧或下方的合同匹配和前后向候选关系。

### 6.2 Word 合同匹配概览

展示每份 Word 合同：

- 合同方向：前向 / 后向。
- 合同名称。
- 合同主体。
- 税率。
- 设备/服务关键词。
- 可能匹配到的宏观合同流。
- 匹配依据。

### 6.3 前向合同 → 天安 → 后向合同多对多候选关系

每行代表：

```text
某前向合同可能通过天安对应到某后向合同
```

展示：

- 置信度。
- 前向合同。
- 后向合同。
- 共同宏观合同流。
- 共同主体。
- 共同设备/服务关键词。
- 税率重合。
- 金额接近线索。
- 背靠背付款线索。
- 状态：待确认 / 已确认 / 已否决。

候选关系必须人工确认后，才能进入设备级拆分。

## 10. 阶段付款释放规则

```text
预付款：
  根据合同签署和预付款条款释放；
  若存在背靠背条款，则需等待上游预付款到账。

到货款：
  根据到货、入库、发票和对应上游回款释放。

安装款：
  根据安装签证、安装数量和对应上游回款释放。

验收款：
  根据验收单、验收数量和对应上游回款释放。

审计款：
  根据业主审计确认数量/金额释放；
  付款基数必须以审计确认金额为准。

质保金：
  根据质保期、遗留问题和最终审计确认金额释放。
```

前向合同和后向合同的付款阶段可能不一致，不能机械一一对应。系统应支持：

```text
前向设备级阶段回款
→ 设备级资金池
→ 后向设备级阶段付款释放
```

## 11. 风险规则

系统应自动识别：

- 前向合同已确认但未能拆到设备。
- 后向合同已确认但找不到对应前向资金来源。
- 同一设备被多个后向合同重复认领。
- 前向已审计但未回款。
- 后向供应商申请付款但对应设备未审计。
- 设备已审计但审计确认金额小于合同金额。
- 前后向税率不一致。
- 前向金额不足覆盖后向付款。
- 已回款金额无法追溯到具体设备。
- 后向阶段付款超过对应设备级可释放金额。
- 付款比例被误识别为税率。
- 合同级已回款但设备级无法证明回款归属。

## 12. 正式化要求

当前原型确认状态如果存在，只能算本地状态。正式系统必须保存：

- 确认人。
- 确认时间。
- 确认依据。
- 来源合同。
- 来源设备清单。
- 来源审计单。
- 来源付款单。
- 来源发票。
- 版本。
- 审计日志。

候选关系在进入资金计算前必须是 `confirmed`。

# 项目审计与优化报告

审计日期：2026-06-11

## 1. 仓库现状

当前仓库是无锡车路云项目管理平台一期原型与文档仓库。主分支 `main` 停留在基础原型提交，`test-pr-flow` 已累计点位管理、地图资产、合同管理、资料管理、运维管理、MAP-XML 转 SVG、导入脚本和本地服务能力。

当前工作区在审计开始时干净，本地分支：

- `main`
- `test-pr-flow`

远端分支：

- `origin/main`
- `origin/test-pr-flow`

## 2. 功能审计

已实现功能覆盖：

- 总览：项目关键指标、功能入口、导入入口。
- 点位管理：表格、卡片、地图三视图，列头多选过滤，列宽拖拽，统计卡片，地图详情抽屉。
- 地图资产：资产扫描、质量检查、预览、导出。
- 设备管理：设备记录和未匹配池展示。
- 合同管理：合同 KPI、前后向合同、宏观合同流、候选关系、设备级资金流、人工确权。
- 资料管理：文档资产分类与扫描结果展示。
- 运维管理：路侧设备状态、异常日历、趋势、地图和本地状态持久化。
- 工具脚本：点位导入、资料扫描、合同关系分析、MAP-XML 转 SVG。

## 3. 主要风险

1. `prototype/app.js` 和 `prototype/styles.css` 体量较大，后续维护成本高。
2. `prototype/server.js` 包含多类接口和大量本机绝对路径，工程化前需要配置化。
3. 合同关系数据和运维状态数据规模较大，主线应区分种子数据、生成数据和运行态数据。
4. 部分合同文件和资料资产包含真实业务文件，推送 GitHub 前应确认仓库可见性、访问权限和保密边界。
5. 当前缺少自动化测试，主要依赖语法检查、接口冒烟和浏览器人工验证。

## 4. 已执行优化

本轮优化聚焦主线合并前的仓库卫生：

- 新增 `.gitignore`，排除系统垃圾文件、运行日志、Python 缓存、本地状态、临时输出和备份文件。
- 将本地运行状态、日志、`.DS_Store`、`__pycache__`、临时输出从 Git 索引移除。
- 新增完整 PRD，统一产品目标、业务对象、功能需求、数据接口和验收标准。
- 保留产品原型、设计文档、导入脚本、地图转换工具和必要兜底数据。

## 5. 建议后续优化

代码层面：

- 拆分 `prototype/app.js`：建议按 `sites`、`mapAssets`、`contracts`、`documents`、`ops`、`sharedMap` 模块切分。
- 拆分 `prototype/styles.css`：建议按全局、导航、表格、地图、合同、运维、主题分层。
- 把 `MAP_ASSET_DIR`、合同根目录、Python 路径迁移到环境变量或本地配置文件。
- 对 MAP-XML 质量检查、合同排除、资料扫描建立单元测试。

数据层面：

- 建立 `sample/`、`seed/`、`runtime/` 三类数据边界。
- 真实合同原件建议转为受控对象存储，Git 仅保存索引、哈希和分类结果。
- 运维每日状态快照不直接进入 Git，保留导入源、样例和统计口径。

产品层面：

- 将 PRD、接口清单、字段字典、ER、页面原型说明保持同一版本号。
- 增加“验收清单”文档，按页面和接口逐项验证。
- 增加合同 OCR、设备级资金流证据链和审计付款释放的二期需求池。

## 6. 合并策略

本轮建议以 `test-pr-flow` 作为事实集成分支合入 `main`，原因：

- `test-pr-flow` 是 `main` 之后所有功能提交的连续线性历史。
- 远端 `origin/test-pr-flow` 已包含主要集成提交，本地 `test-pr-flow` 只比远端多一个异常 NodeID 闪烁开关提交。
- 本轮新增清理和 PRD 提交后，可快进或普通合并到 `main`，再推送 `main` 和 `test-pr-flow`。

## 2. 核心结论

合同管理不能按普通“合同台账”设计。应设计成：

```text
合同流图谱
→ 分项包
→ 合同/采购/销售台账
→ 设备级合同明细
→ 设备级前后向合同归属
→ 点位设备安装关联
→ 设备级到货/入库/领料/安装/验收/付款履约
→ 设备级资金流与业主回款联动
```

系统必须支持从任意对象双向追溯：

```text
点位 → 设备 → 合同明细 → 前向合同/后向合同 → 合同流链路 → 上游/下游单位 → 设备级金额/付款/风险

合同 → 分项包 → 设备明细 → 点位安装位置 → 到货/入库/安装/验收/付款节点时间
```

## 3. 合同流数据理解

`合同流` sheet 的主要字段包括：

| 字段 | 含义 |
|---|---|
| 序号 | 分项编号，如 1.1、1.8、1.9.1 |
| 项目名称 | 分项或拆分项名称 |
| 分项小计 | 对应分项金额 |
| 投标价（分项） | 对外投标或上游流转金额 |
| 拆分包名称 | 合同拆包或实际流转包名称 |
| 拆分包价格 | 拆分后的价格 |
| 一级分包商至六级分包商 | 合同链路节点 |
| 一级分包价格至六级分包价格 | 每一级流转价格 |
| 原厂价格分项小计 | 原厂/最终供应价格 |
| 毛利 | 分项或链路毛利 |
| 折扣率 | 毛利或折扣比例 |
| 供应商 | 主要供应商 |
| 备注 | 对应分项明细 sheet |

## 6. 合同流图谱

### 6.1 页面目标

将 `合同流` sheet 中的多级分包商拆成可视化链路，支持查看：

- 某分项从移动或天安流向哪些单位。
- 哪些链路流回天安。
- 天安继续向哪些供应商流出。
- 每条边的金额、价差、毛利和风险。

### 6.2 链路建模

系统将每一行合同流拆成：

```text
合同流记录 contract_flow_line
→ 合同流节点 contract_flow_node
→ 合同流边 contract_flow_edge
```

例如：

```text
移动 → 浪潮 → 天安 → 大为 → 大华
```

拆成 4 条边：

```text
移动 → 浪潮
浪潮 → 天安
天安 → 大为
大为 → 大华
```

### 6.3 链路类型

| 类型 | 说明 |
|---|---|
| mobile_outbound | 从移动向后流动 |
| tianan_outbound | 从天安直接向后流动 |
| mobile_to_tianan | 移动链路中流回天安 |
| tianan_downstream | 天安继续向后流动 |
| self_owned | 移动自有、天安自有或集成费；其中天安自有/集成费仍需判断是否要通过采购数据回流业主车联网集团 |
| suspended | 暂时不实施 |
| service_only | 软件、服务、运维等非硬件实物 |

### 9.2 合同明细类型

| 类型 | 说明 |
|---|---|
| hardware_device | 硬件设备 |
| software_product | 软件产品 |
| custom_development | 定制开发 |
| service | 运维、租赁、测评、安装等服务 |
| construction | 施工或安装工程 |
| network_resource | 物联网卡、骨干光纤等资源 |
| placeholder | 暂时不实施或预留项 |

### 9.3 设备级管理要求

设备级合同明细必须支持：

- 记录合同链路。
- 记录每一条设备/服务明细对应的前向合同主体、后向合同主体、最终供应商/原厂。
- 记录投标价、采购价、原厂价、毛利。
- 关联设备安装位置表中的 `物料名称`、`品牌型号`、`NodeID`、`安装位置`。
- 关联点位。
- 关联入库、出库、领料、安装、验收数量。
- 识别合同数量与实际设备数量差异。
- 支持同一个分项 sheet 内不同设备分属不同前向/后向合同主体。
- 支持同一合同内不同设备的到货、入库、安装、验收、付款节点发生时间不同。

### 9.4 设备级前后向合同归属

各分项 sheet 不能只按 sheet 名或分项包统一确定合同主体。真实情况是：同一个子模块、同一个分项 sheet 下，不同设备可能对应不同的前向销售合同、后向采购合同、原厂供应商和资金路径。

因此合同明细必须采用“行级合同归属”：

```text
分项 sheet 行
→ 合同明细 contract_item
→ 前向合同归属 contract_item_contract_role(role=front_sales)
→ 后向合同归属 contract_item_contract_role(role=back_purchase)
→ 原厂/最终供应 contract_item_contract_role(role=original_supplier)
```

字段要求：

- 前向合同主体：如浪潮、上汽帆一、中通服网盈、工业安装等。
- 后向合同主体：如万集、航天海特、华通、海康智联等。
- 天安角色：销售方、采购方、集成方、自有交付方。
- 合同文件：关联前项销售 PDF 或后项采购 PDF。
- 合同金额分摊：支持按数量、按金额、按人工确认分摊。
- 设备范围：支持关联多个点位、多个安装位置、多个 SN。

## 10. 前向销售与后向采购合同文件管理

### 10.1 前向天安销售合同

### 11.2 匹配逻辑

自动匹配维度：

1. 合同明细 `分项名称` ↔ 设备表 `物料名称`
2. 合同明细 `品牌及型号` ↔ 设备表 `品牌型号`

5. 分项编号 ↔ 设备所属合同板块

匹配结果：

- 自动匹配
- 疑似匹配
- 人工匹配
- 匹配失败
- 不需要匹配

## 12. 设备级合同履约、资金流与付款

### 12.1 设备级履约状态

合同履约不只看合同状态，还要看设备级过程：

```text
已签约
→ 已下单
→ 已送货
→ 已入库
→ 已开箱验收
→ 已领料
→ 已安装
→ 已点位验收
→ 已付款
→ 已归档
```

同一合同中的不同设备，可能分批到货、分批入库、分批安装、分批验收，也可能触发不同批次付款。因此系统必须在设备级记录每一个节点的发生时间。

设备级履约事件：

- 合同明细确认
- 下单
- 送货
- 到货查验
- 入库
- 开箱验收
- 领料
- 安装
- 点位初验
- 试运行
- 终验
- 审计确认
- 质保开始
- 质保结束

### 12.2 资金流动

资金流动必须细化到设备/合同明细级。系统需要同时管理：

```text
业主/车联网集团 → 移动/浪潮/上汽帆一等前向主体 → 天安
天安 → 后向供应商/施工单位/服务商
```

资金流字段：

- 资金方向：收入、支出、回流、代付、暂估、质保金。
- 关联设备/合同明细。
- 关联前向合同。
- 关联后向合同。
- 付款触发事件。
- 应收金额、实收金额、应付金额、实付金额。
- 上游回款比例。
- 后向付款比例。
- 是否背靠背付款。
- 是否需要回流业主车联网集团。

天安自有或集成费处理规则：

- 不能简单标记为 `self_owned` 后结束。
- 其中一部分需要通过采购数据、设备清单、服务确认或审计结果回流到业主车联网集团。
- 系统需标记 `owner_backflow_required`，并追踪回流依据、回流金额、回流状态和对应业主确认材料。

### 12.3 付款节点

付款节点支持按合同、分项包、设备明细维度绑定：

- 合同签署款
- 到货款
- 入库验收款
- 安装完成款
- 初验款
- 终验款
- 质保金

付款触发条件可配置：

- 合同签署完成
- 到货数量达到合同数量比例
- 入库数量达到合同数量比例
- 安装数量达到合同数量比例
- 点位验收通过
- 发票已到
- 上游回款已到账
- 业主或甲方客户已完成同阶段验收

### 12.4 背靠背付款规则

后向采购合同普遍存在背靠背付款逻辑：甲方向乙方付款以业主或甲方客户向甲方支付相应等比例款项为前提。

系统应支持：

- 付款节点绑定上游回款节点。
- 若上游未回款，后向付款节点自动标记为“待上游回款”。
- 若设备已验收但上游未回款，形成资金风险。
- 若上游部分回款，按设备/明细级分摊可付款额度。

## 13. 合同风险与数据治理

系统应自动识别：

- 合同链路断裂：有拆分包但缺少上游或下游单位。
- 合同金额不一致：投标价、拆分包价格、各级价格、原厂价格无法对平。
- 设备未匹配合同：设备安装位置表中有设备但无法匹配合同明细。
- 合同未匹配设备：合同明细为硬件但没有实际设备/点位关联。
- 数量差异：合同数量、送货数量、入库数量、领料数量、安装数量不一致。
- 价格倒挂：原厂采购价高于流转价格。
- 毛利异常：毛利为负或折扣率异常。
- 签约延期：合同签署计划逾期。
- 到货影响项目进度：合同未签署但设备依赖到货。
- 前向回款未到账导致后向付款阻塞。
- 同一合同下设备节点时间不一致但未拆分履约批次。
- 天安自有/集成费未标记是否需回流业主车联网集团。
- 同一分项 sheet 内设备行错误继承 sheet 级合同主体。

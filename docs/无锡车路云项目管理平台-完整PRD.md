# 无锡车路云项目管理平台完整 PRD

版本：2026-06-11 主线整合版

## 1. 产品背景

无锡车路云一体化项目同时涉及点位规划、路侧设备、地图资产、合同流、仓储交付、资料归档、运维状态和审计付款等多条业务线。现有数据分散在点位规划表、设备安装位置表、MAP/RSI/信号机配置文件、合同文件、运维状态表和资料目录中，项目管理需要一个以 `NodeID` 为主线的统一工作台。

本平台定位为一期原型与数据治理工作台，用于验证业务对象、页面流程、数据口径和导入治理规则，为后续工程化系统建设提供产品、数据、接口和原型依据。

## 2. 产品目标

1. 建立以 `NodeID` 为核心的点位台账，支持表格、卡片、地图三种工作视图。
2. 将设备安装位置、供应商、合同、入库、领料、安装和验收状态关联到点位。
3. 建立 MAP、RSI、信号机、SVG 预览等地图资产的完整性扫描与导出能力。
4. 建立合同管理工作台，支持前向销售、后向采购、宏观合同流、前后向候选关系、设备级资金流和业主审计付款。
5. 建立资料资产管理能力，支持合同、会议、变更、施工、物资、地图、运维等资料分类和业务对象关联。
6. 建立运维状态快照能力，支持离线/异常设备统计、日历、趋势和地图联动。
7. 建立数据导入治理闭环，保留原始字段、错误行、未匹配池和人工确权记录。

## 3. 用户与场景

核心用户包括项目管理、工程建设、交付实施、运维、合同商务、资料管理和管理层。

典型场景：

- 项目经理查看点位建设、设备落位、合同流和运维风险总览。
- 工程人员按区域、类型、信号机厂商、感知类型筛选点位，核对点位和设备。
- 地图数据人员检查 MAP-XML/MAP-JSON/RSI/信号机文件是否齐套，定位上下游 NodeID、停止线等质量问题。
- 商务人员维护前向/后向合同、清单、税率、付款节点和前后向关系候选。
- 资料人员扫描和归档合同、会议、施工、物资、地图和运维文件。
- 运维人员导入每日路侧设备运行状态表，查看离线设备、异常日历和趋势。

## 4. 信息架构

主导航包含：

- 总览
- 点位管理
- 设备管理
- 地图资产
- 导入中心
- 坐标异常
- 合同管理
- 出入库
- 资料管理
- 运维管理

所有功能以 `NodeID`、合同编号、资料文件哈希、设备编码等稳定标识进行关联。点位页面是空间主线，合同页面是资金主线，资料页面是证据主线，运维页面是状态主线。

## 5. 核心业务对象

### 5.1 点位

点位以 `NodeID` 为主键，保留 `CrossID`、点位名称、英文名称、行政区域、点位类型、信号机厂商、感知点位类型、自适应、可变车道、原始 CGCS2000 坐标和业务展示 GCJ-02 坐标。

点位类型覆盖路口、路段、R1、R2、R3、V2P、礼让行人、匝道汇入、匝道汇出、公交、市政作业等业务口径。

### 5.2 设备安装位置

设备安装位置通过 `NodeID + 安装位置 + 物料编码/物料名称` 关联点位。核心字段包括供应商、品牌型号、合同名称、送货、入库、开箱验收、领料、安装数量和单据索引。

### 5.3 地图资产

地图资产按 `NodeID` 聚合 MAP-XML、MAP-JSON、RSI-XML、RSI-JSON、信号机 Excel、SVG 预览等文件。系统扫描文件目录并生成完整性、版本、预览和导出结果。

质量检查包括：

- MAP、RSI、信号机、SVG 是否齐套。
- MAP-XML/MAP-JSON 上下游 NodeID 是否存在。
- MAP-XML 上下游 NodeID 数字位数是否小于 5。
- 正常机动车道是否存在停止线。

### 5.4 合同

合同对象分为前向销售和后向采购。系统应支持合同文件、合同清单、宏观合同流、前后向候选关系、设备级资金流单元、税率、付款阶段、审计状态和人工确权。

税率是合同与设备资金流匹配的重要字段，应作为匹配、校验和风险提示的一等字段。

### 5.5 资料资产

资料文件按合同、会议、变更、计划、物资、施工、地图、运维、管理、未分类等类别管理。文件记录应包含路径、文件名、大小、修改时间、SHA256、业务对象、分类、来源批次和解析状态。

### 5.6 运维状态

运维状态来自每日路侧设备运行状态表，不得与设备安装位置表混用。运维状态通过路口 ID 关联点位 `NodeID`，坐标以点位表为准。

## 6. 功能需求

### 6.1 总览

总览展示项目点位、设备记录、未匹配设备、合同流、地图资产、资料资产和运维状态摘要。核心卡片支持跳转到对应功能页。总览提供点位表、设备表等导入入口。

### 6.2 点位管理

点位管理提供表格、卡片、地图三种视图，三种视图共享筛选条件。

表格视图：

- 首列为序号列。
- 表头完全不透明。
- 表格区域满铺到页面底部。
- 列宽支持手动拖拽调整，标题栏显示纵向拖拽标线。
- 表头过滤与列标题占用一行空间。
- 区域、类型、信号机厂商、感知点位类型、自适应、可变车道支持表头多选过滤。
- 过滤入口为白色磨砂/白瓷圆形按钮，显示“筛”或“筛N”。
- 未选择任何项表示不过滤该列。

卡片视图：

- 页面水平方向最多布置 5 个卡片。
- 卡片标题第一行展示 `NodeID + 点位名称`，高亮显示。
- 区域徽章显示在右上角。
- 去除“施工中”“已规划”等状态徽章。
- macOS 风格下避免黑底白标签的强刺激配色。

地图视图：

- 地图高度铺满到页面底部。
- 右侧点位列表卡片高度与地图区域一致。
- 支持点位点击打开详情抽屉。
- 地图内提供“上下游 NodeID 异常点位闪烁”滑动开关，默认关闭。
- 开关打开时，MAP-XML 上下游 NodeID 位数小于 5 的点位闪烁；关闭时不闪烁。

点位详情抽屉：

- 宽度为原详情页约 2 倍。
- 左侧显示该点位 MAP-XML 转换的 SVG 高精地图，SVG 预览为正方形，底部可铺到弹出页底部。
- 右侧上方显示高德地图。
- GCJ-02 经纬度显示在高德地图上方副标题。
- 原信息卡片缩小并显示在高德地图下方。
- 下方每个信息卡片内文字只占一行。

### 6.3 设备管理

设备管理按设备类型、安装位置、供应商、合同、履约状态穿透展示。支持未匹配池治理，将无法通过 `NodeID` 匹配的设备、车端、机房或服务项纳入人工治理流程。

### 6.4 地图资产

地图资产页应提供：

- 地图资产目录扫描。
- 按 `NodeID` 聚合地图文件。
- 完整性、版本、文件类别和预览状态统计。
- 列头多选过滤。
- MAP-XML/MAP-JSON SVG 预览。
- 单点位资产 ZIP 导出。
- 资产清单 XLSX 导出。
- MAP-XML 上下游 NodeID 位数异常、上下游缺失、停止线缺失等质量提示。

### 6.5 导入中心

导入中心用于展示点位、设备、合同、资料、运维等导入任务的阶段、错误、警告和治理结果。导入必须保留原始表头和审计字段，不得因前端展示需要破坏源表结构。

### 6.6 坐标异常

坐标异常页展示 CGCS2000/WGS84 到 GCJ-02 转换异常、缺失、越界或无法落图的数据。业务页面默认使用 GCJ-02 坐标，原始坐标用于审计。

### 6.7 合同管理

合同管理应包含：

- 合同 KPI。
- 前向销售合同池。
- 后向采购合同池。
- 宏观合同流骨架。
- 前后向合同候选关系。
- 合同清单行。
- 设备级资金流单元。
- 税率核对。
- 审计付款释放。
- 人工确权、排除和恢复。
- 合同文件导入、扫描、打开和移出系统。

合同管理不得仅依赖文件名推断设备级关系；扫描件应先做可抽取性分诊，必要时进入 OCR 流程。

### 6.8 出入库

出入库模块围绕送货、入库、开箱验收、领料、安装数量进行台账展示，并与设备安装位置、合同清单和付款节点联动。

### 6.9 资料管理

资料管理用于展示资料资产分类、文件数量、业务对象关联和批次扫描结果。资料原件进入 `document_assets/raw` 或对象存储；前端和数据库保存索引、哈希、版本、业务关系、解析结果和审计信息。

### 6.10 运维管理

运维管理支持：

- 导入每日路侧设备运行状态表。
- 保存历史状态快照。
- 展示在线、离线、异常、未匹配状态。
- 通过异常日历查看每日状态。
- 离线/异常设备地图。
- 离线/异常趋势图。
- 与点位、设备和供应商维度联动。

## 7. 数据与接口

当前原型为静态前端加 Node.js 本地服务。

核心数据文件：

- `prototype/data.js`：点位、设备和基础原型数据。
- `prototype/roadside-status.js`：运维状态初始数据。
- `prototype/data/document-assets.json`：资料资产前端兜底数据。
- `prototype/data/contract-relationships.json`：合同关系前端兜底数据。
- `document_assets/import_batches/latest-document-scan.json`：资料扫描批次结果。

本地服务接口：

- `GET /api/health`
- `GET/POST /api/roadside-status-state`
- `GET /api/map-assets`
- `GET /api/map-assets/file`
- `GET /api/map-assets/export`
- `GET /api/contract-file`
- `POST /api/contract-file/open`
- `POST /api/contract-import`
- `POST /api/contract-system-remove`
- `POST /api/contract-relationships/rebuild`
- `GET /api/document-assets`

## 8. 非功能需求

- 原型可通过 `node prototype/server.js` 在本地运行。
- 页面在默认、命令、轨迹等视觉主题下保持可读。
- 表头、弹层、地图抽屉、卡片和表格布局应避免遮挡、重叠和透明度过高。
- 大型生成数据应区分产品兜底数据与本地运行态数据；日志、缓存、`.DS_Store`、`__pycache__` 不进入 Git。
- 文件扫描、合同导入、地图资产扫描应有错误兜底，不因单个文件失败导致页面不可用。

## 9. 验收标准

1. 访问 `http://127.0.0.1:4173` 可打开完整原型。
2. 十个主导航页面均可切换，页面标题、副标题和内容一致。
3. 点位管理三视图筛选条件一致，表格列头过滤、多选、列宽拖拽可用。
4. 地图视图异常 NodeID 闪烁开关默认关闭，打开后仅异常点位闪烁。
5. 点位详情抽屉可展示高德地图、GCJ-02 坐标和 MAP-XML SVG 预览。
6. 地图资产接口可返回资产聚合、完整性统计和质量标记。
7. 合同管理可展示前向、后向、候选关系、人工确权和审计付款信息。
8. 资料管理可读取资料资产扫描结果。
9. 运维管理可读取初始状态并通过本地服务保存历史状态。
10. 仓库主线不包含本地日志、缓存、系统垃圾文件和临时输出。

## 10. 后续工程化建议

- 将 `prototype/app.js` 拆分为页面模块、数据访问模块、地图模块和合同模块。
- 将本地绝对路径改为配置文件或环境变量。
- 将大型 JSON 兜底数据迁移到可版本化的种子数据或对象存储索引。
- 建立 Playwright 冒烟测试，覆盖主导航、点位三视图、地图资产接口、合同管理和运维状态保存。
- 为 MAP-XML/MAP-JSON 质量检查补充单元测试。
- 建立合同 OCR 与设备级资金流的证据链审核工作流。

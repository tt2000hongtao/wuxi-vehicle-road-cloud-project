## 2. 点位管理接口

### 2.1 点位列表查询

```text
GET /api/v1/sites
```

用途：表格视图、卡片视图共用。点位管理页面不分页展示，前端使用右侧滚动条浏览；后端可按性能需要支持游标或滚动加载。

请求参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| nodeId | string | 否 | NodeID |
| siteName | string | 否 | 点位名称 |
| district | string | 否 | 行政区域 |
| originalDistrict | string | 否 | 原始行政区域，如经开 |
| siteType | string | 否 | 点位类型 |
| signalVendor | string | 否 | 信号机厂商 |
| perceptionSiteType | string | 否 | 感知点位类型 |
| isAdaptiveSite | boolean | 否 | 是否自适应点位 |
| isVariableLaneSite | boolean | 否 | 是否可变车道路口 |
| hasIssue | boolean | 否 | 是否有异常 |
| sortIssueFirst | boolean | 否 | 是否将异常点位排在前面，默认 true |

说明：

- 点位总数按有效点位行数统计，不按 Excel 物理行数统计。
- NodeID 为空、NodeID 重复或关键点位信息不足的导入行不得进入正式点位库，应进入导入错误行或待补全治理清单。
- 经开区在项目过程中已并入新吴区；接口筛选 `district` 使用归并后的标准行政区域，`originalDistrict` 仅用于审计和回查。

返回字段：

- id
- nodeId
- crossId
- siteName
- nodeNameEn
- district
- originalDistrict
- districtNormalizeRule
- siteType
- signalVendor
- perceptionSiteType
- isAdaptiveSite
- isVariableLaneSite
- longitudeGcj02
- latitudeGcj02
- deviceRecordCount
- deviceTypeCount
- installedQuantity
- issueCount
- updatedAt

### 2.1A 删除点位

```text
DELETE /api/v1/sites/{siteId}
```

用途：逻辑删除点位。

删除规则：

- 删除前必须校验点位下是否存在 matched `device_installation_item`。
- 若存在下挂设备，返回 `SITE_HAS_DEVICE_ITEMS`，提示用户先到设备管理删除或迁移该点位下设备。
- 删除成功时写入 `deleted_at`、`deleted_by`、`delete_reason`，并刷新周边点位缓存。

请求体：

```json
{
  "deleteReason": "重复点位清理"
}
```

返回字段：

- id
- deletedAt
- deletedBy

### 2.4 保存点位

```text
PUT /api/v1/sites/{siteId}
```

用途：保存点位编辑页基础信息。

请求体：

```json
{
  "siteType": "匝道汇入",
  "district": "锡山",
  "originalDistrict": "锡山",
  "nodeId": "10372",
  "crossId": "-1",
  "nodeNameEn": "TaiHuDaDaoBei~YouYiNanLuXi",
  "siteName": "太湖大道北~友谊南路西",
  "signalVendor": "华通",
  "perceptionSiteType": "雷视融合",
  "isAdaptiveSite": false,
  "isVariableLaneSite": false,
  "variableLaneCount": null,
  "variableLaneDirection": ""

```

规则：

- NodeID 必须唯一。
- 点位编辑页面默认不编辑或展示原始 CGCS2000 坐标；如需调整坐标，应通过坐标治理流程刷新 GCJ-02 坐标。
- GCJ-02 坐标变化后刷新周边点位缓存。

## 11. 文档资产管理接口

文档资产接口遵循“元数据查询、原件下载、业务对象关联、批量导入、解析质检分离”的原则。大文件不进入前端内嵌数据，前端只展示后端返回的元数据和预览地址。

### 11.1 文档上传登记

```text
POST /api/v1/documents/upload
```

用途：上传或登记单个文件，计算 `sha256`，写入文档元数据，并可同时建立业务对象关联。

请求体采用 `multipart/form-data`：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| file | file | 是 | 文件原件 |
| projectCode | string | 是 | 项目编码 |
| documentCategory | string | 是 | 文档分类 |
| documentSubcategory | string | 否 | 文档子类 |
| objectType | string | 否 | 关联对象类型 |
| objectId | string | 否 | 关联对象 ID，例如 NodeID、contractId |
| uploadedBy | string | 否 | 上传人 |

返回字段：

- id
- documentNo
- fileName
- fileExt
- fileSizeBytes
- sha256
- storageProvider
- storageKey
- currentVersionNo
- reviewStatus
- parseStatus

### 11.2 文档列表查询

```text
GET /api/v1/documents
```

请求参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| projectCode | string | 否 | 项目编码 |
| keyword | string | 否 | 文件名、编号、摘要关键字 |
| documentCategory | string | 否 | 文档分类 |
| objectType | string | 否 | 关联对象类型 |
| objectId | string | 否 | 关联对象 ID |
| reviewStatus | string | 否 | draft/effective/archived/voided/pending_review |
| parseStatus | string | 否 | pending/success/failed/skipped |
| qualityStatus | string | 否 | normal/warning/error |
| pageNo | int | 否 | 页码 |
| pageSize | int | 否 | 每页数量 |

返回字段：

- id
- documentNo
- fileName
- documentCategory
- documentSubcategory
- fileExt
- fileSizeBytes
- currentVersionNo
- reviewStatus
- archiveStatus
- parseStatus
